<objective>
## Goal
Build the Experiment stage UI: create experiments by selecting a test set + RAG config + metrics, list experiments with status, and run them with live SSE progress tracking.

## Purpose
Connects the Test stage (approved questions) to evaluation — users can now configure and execute experiments against their RAG pipeline, seeing real-time progress as each question is evaluated.

## Output
- Experiment API client functions in `lib/api.ts`
- ExperimentCreate form component (test set + RAG config + metrics selection)
- ExperimentList component (browse experiments with status badges)
- ExperimentRunner component (SSE streaming progress UI)
- ExperimentPage assembled with all components
</objective>

<acceptance_criteria>

## AC-1: Experiment Creation
```gherkin
Given a project with at least one test set and one RAG config
When the user selects a test set, RAG config, enters a name, and optionally picks metrics
Then a new experiment is created with status "pending" and appears in the experiment list
```

## AC-1b: Experiment Creation Error Handling <!-- audit-added -->
```gherkin
Given the user submits the create experiment form
When the backend returns an error (422 no approved questions, 422 invalid config, network failure)
Then the error message is displayed inline near the form
And the form remains populated so the user can correct and retry
And no experiment is added to the list
```

## AC-2: Experiment List with Status
```gherkin
Given a project with experiments in various states (pending, running, completed, failed)
When the user views the Experiment stage
Then all experiments display with name, test set, RAG config, status badge, and creation date
And status badges use color coding (pending=yellow, running=blue, completed=green, failed=red)
```

## AC-3: Live Experiment Runner
```gherkin
Given a pending experiment
When the user clicks "Run" and selects metrics
Then an SSE connection streams progress events showing current/total questions
And a progress bar updates in real-time
And on completion the experiment status updates to "completed"
And on error the experiment status updates to "failed" with error message
```

## AC-3b: SSE Connection Failure Handling <!-- audit-added -->
```gherkin
Given an experiment is running with active SSE connection
When the network connection drops or the fetch stream errors
Then the user sees a connection-lost message with the last known progress
And the experiment list can be refreshed to check actual backend status
And no stale "running" state persists in the UI after page refresh
```

## AC-3c: Form Validation <!-- audit-added -->
```gherkin
Given the user is filling the create experiment form
When they attempt to submit with empty name, no test set selected, or no RAG config selected
Then inline validation prevents submission and highlights the missing field
```

  <name>Task 1: Experiment API client and types</name>
  <files>frontend/src/lib/api.ts</files>
  <action>
    Add TypeScript interfaces and API functions for experiments:

    **Interfaces to add:**
    - `Experiment` — id, project_id, test_set_id, name, model, model_params, retrieval_config, chunk_config_id, embedding_config_id, rag_config_id, baseline_experiment_id, status ('pending'|'running'|'completed'|'failed'), started_at, completed_at, created_at, plus optional aggregate fields (test_set_name, rag_config_name, approved_question_count, result_count, aggregate_metrics)
    - `ExperimentCreate` — test_set_id, rag_config_id, name
    - `ExperimentResult` — id, test_question_id, question, reference_answer, question_type, persona, response, retrieved_contexts ({content, chunk_id?}[]), metrics (Record<string, number>), metadata, created_at
    - `RunExperimentRequest` — metrics?: string[]

    **API functions to add:**
    - `fetchExperiments(projectId)` → Experiment[]
    - `createExperiment(projectId, data: ExperimentCreate)` → Experiment
    - `fetchExperiment(projectId, experimentId)` → Experiment (single with aggregates)
    - `deleteExperiment(projectId, experimentId)` → void
    - `fetchExperimentResults(projectId, experimentId)` → ExperimentResult[]

    **SSE helper function:**
    - `runExperimentSSE(projectId, experimentId, metrics?: string[])` → returns EventSource-like interface
    - This should POST to `/api/projects/${projectId}/experiments/${experimentId}/run` with metrics body
    - Since SSE typically uses GET but backend uses POST, use fetch with streaming response reader instead of EventSource
    - Parse SSE text stream: split on `\n\n` boundaries, parse `event:` and `data:` fields from each chunk. Handle partial chunks across read boundaries. <!-- audit-added: explicit parsing spec -->
    - Parse SSE events: "started", "progress", "completed", "error"
    - Return an object with `onStarted`, `onProgress`, `onCompleted`, `onError`, `onConnectionError` callbacks and `abort()` method <!-- audit-added: onConnectionError -->
    - `onConnectionError` fires when fetch stream throws (network drop, abort) — passes last known state so UI can show partial progress <!-- audit-added -->

    Follow existing patterns in api.ts — use the `request<T>` wrapper for standard CRUD, custom fetch for SSE streaming.
  </action>
